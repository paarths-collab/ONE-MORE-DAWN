# Scenario Matrix — expected behavior (QA checklist)

> One row per situation. **Trigger condition** cites the real constant; **Expected game
> behavior** is what the resolver / lazy loop / route actually does today; **Where enforced**
> is the file that owns the rule. Rows marked `[FINDING]` are places the real code differs
> from the intuitive expectation — read those carefully during QA.
>
> Constants live in `src/shared/balance.ts`; resolution logic in
> `src/server/game/resolver.ts`; the day loop in `src/server/game/lazyResolve.ts`.

| Situation | Trigger condition (real threshold) | Expected game behavior | Where enforced (file) |
|---|---|---|---|
| **Food hits 0 after consumption** | Stock goes negative after `food -= consumed` (consumed = `ceil(population*0.15 + activePlayers*0.5)` × law × trait mults) | Hunger: `deaths = ceil(missingFood * 0.3)` (`BALANCE.hunger.deathsPerMissingFood`), `morale -8` (`BALANCE.hunger.moralePenalty`), food clamped to `0`. Event: "Hunger swept the city…". | `resolver.ts` §3 |
| **Power below 25** | `power < BALANCE.lowPowerThreshold` (=25) after production/decay | `morale -4` (`BALANCE.lowPowerMoralePenalty`); event "The lights flicker…" → dashboard low-power flicker. | `resolver.ts` §3 |
| **Medicine below 10** | `medicine < BALANCE.sickness.threshold` (=10) | If `medicine >= 2`: spend `-2` (`medicineCostPerDay`). If `medicine < 2`: `population -2` (`deathsIfNone`), event "Sickness claimed 2 lives…". At/above 10 it still pays the `-2` upkeep. | `resolver.ts` §3 |
| **Morale below 15** | `morale < BALANCE.morale.collapseThreshold` (=15) | Desertion: `population -3` (`desertersPerDay`); event "3 citizens slipped away…". | `resolver.ts` §3 |
| **Threat reaches 100** | `next.threat >= BALANCE.raid.triggerThreshold` (=100) **after clamp** | Red Signal raid: `food -20, power -15, morale -15, population -8`; `threat` reset to `40` (`postRaidThreat`). Each `guard_wall` action that day dampens **every** loss by `3` (`guardDampenPerAction`), floored at 0. Raid loss is applied **before** the fall check, so it can fell the city. | `resolver.ts` §4b |
| **Population ≤ 10** | `next.population <= BALANCE.fall.populationThreshold` (=10) | `status = 'fallen'`; no next crisis / no law picked; epitaph event pushed (raid speaks its own line). | `resolver.ts` §4c |
| **Nobody votes on the crisis** | `winningOption(votes)` returns `null` (no option with count > 0) | No crisis effect applied; event "Crisis \"…\": nobody voted. The moment passed unanswered." | `resolver.ts` §2 |
| **Tie crisis vote** | Two+ options share the top count | Deterministic tie-break: sort by **count desc, then optionId asc** (`nB - nA \|\| idA.localeCompare(idB)`) → lowest id wins (e.g. `a` beats `b`). | `resolver.ts` `winningOption` |
| **Nobody plays for N days** | `lastResolvedDate !== today` after a gap; `status === 'alive'` | Lazy catch-up resolves **exactly ONE** missed day (the whole loop is a single `resolveDay` call, not a per-day replay). Next visit resolves the next one. Missed days are compressed, not iterated. | `lazyResolve.ts` `runLazyResolution` |
| **City fallen → visited later** | `city.status !== 'alive'` | `runLazyResolution` early-returns `{ resolving: false }` **without advancing**; day never increments. Only memorial (timeline stays) + mod `/menu/reset` can move it. | `lazyResolve.ts` (guards at needsResolve + inside lock) |
| **Same Reddit user in two subreddits** | Each installation has its own Redis namespace + `worldSeed = hash(subredditId)` | Two **separate** cities, day counters, and player profiles (Devvit Redis is installation-scoped; keys like `city:state`, `players`, `day:{d}:*` are per-install). Same human plays two independent games. | `redisKeys.ts`, `api.ts` `deriveWorldSeed` |
| **User double-taps an action** | Two `/action` (or `/mission/start`) requests race the `players` watch | Optimistic-concurrency: both `watch(KEYS.players)`; the loser's `exec` returns empty/throws → 409 "Busy — try again". Only **one** energy spend + one aggregate increment counts. | `api.ts` `/action`, `mission.ts` `/start` (`execOrConflict`) |
| **Double-vote / double-plan / double-pledge** | Second request same day | Per-day voter/pledger hash acts as the lock: existing entry → 409 ("You already voted today." / "…backed a plan today." / "…already pledged today."). One tally increment max. | `api.ts` `/vote`, `/strategy`, `/pledge` |
| **Mission token reuse / expired / spoofed loot** | `evaluateMission` gate fails | Server regenerates map from seed and re-computes loot; rejects (400/404/409): consumed, wrong user, stale day, expired (TTL 10min), too many/duplicate/unknown crates, too-fast (`<5000ms` or below `minFeasibleMs`), too-slow. Reuse also blocked by watch+del on the token key. | `missionRules.ts` `evaluateMission`, `mission.ts` `/complete` |
| **Mod force-resolve** | `POST /menu/force-resolve`, city `alive` | Resolves current `city.day` once: exactly **one** timeline entry appended, `day+1`, snapshot + marked outcome + `lastResolvedDate` set. Toast reports old→new day + status. **`[FINDING]`** passes `roleCounts:{}` and `activeUserCount:0`, so it skips speaker-morale and active-player scarcity scaling — a forced dawn is slightly gentler than an organic one. Fallen city → toast, no-op. | `menu.ts` `/force-resolve` |
| **Mod reset** | `POST /menu/reset` (any state) | New cycle: deletes `city:state`, `timeline`, `city:history`, `players`, both leaderboards, `marked:outcomes`, and every `day:{1..lastDay+1}:*` key (batched ≤100). Writes `newCityState(cycle+1)` → `day=1`, `cycle+1`, fresh trait roll. Toast "City reset. Cycle N, day 1." | `menu.ts` `/reset` |
| **Marked goal met at dawn** | `markedPledged >= marked.goal` (goal = `min(200, 10 + ceil(yesterdayActives*4))`) | `morale +4` (`savedMoraleBonus`); `{ name, saved:true }` persisted → surfaces as `savedYesterday`; event "…was saved…". | `resolver.ts` §2c, `marked.ts` |
| **Marked goal missed at dawn** | `markedPledged < marked.goal` | `morale -3` (`lostMoralePenalty` — "a memorial, not a wipe"); `{ name, saved:false }` recorded; event "Memorial: … was lost at dawn…". | `resolver.ts` §2c |
| **Empty city / everyone offline** | All aggregates empty; `activeUserCount = 0` | Resolver still runs: passive decay applies (`threat +6`, `power -3`), consumption still charges `ceil(population*0.15)`, crisis "moment passed", Marked missed → `morale -3`. Day advances. City can decay toward fall with zero players. | `resolver.ts`, `lazyResolve.ts` |
| **Extreme resource values (near caps)** | Production/effects push a vital past its bound | Clamped at write: `food = min(300, max(0, round))` (`foodStoreCap`), `medicine = min(120, …)` (`medicineStoreCap`), `power/morale/threat/defense` = `clampPct` (0..100), `population = max(0, round)`. Raid re-clamps after its losses. | `resolver.ts` §4 (`clampPct`, `clampStock`) |
| **Crisis eligibility gated by vitals** | `ration_riots` needs `food <= 15`; `sickness_spreads` needs `morale <= 70`; several need `city.day+1 >= minDay` | `pickNextCrisis` filters by `isEligible`; ineligible crises are never selected. Same crisis never repeats two days running. Empty pool → `strange_signal`/`first_light` fallback. | `crises.ts` `isEligible`, `pickNextCrisis` |
| **Faction winner sets tomorrow's law** | Highest `factionInfluence` (ties by `builders>wardens>seekers>hearth`) on an `alive` city | `activeLaw = winner`, `lawExpiresDay = next.day` (lifespan 1). No winner → carry unexpired law or clear. Fallen cities get no law. | `resolver.ts` §5b, `winningFaction` |
| **Council unity bonus** | Winning plan has `>= 3` voters (`minPlanVoters`, back_council pledges count toward quorum) **and** aligned actions ÷ total effort `>= 0.6` (`alignedShareThreshold`) | `morale +5` (`unity.moraleBonus`) applied **before** consumption so it participates in the collapse check. | `resolver.ts` §2b |
